★サービスマネジメント概要

ITIL(Information Texhnology Infrastructure Library) => ITサービスマネジメントのペストプラクティス書籍の一部

・ITIL成功理由
- ベンダに依存しない
- 規範的ではない
- ベストプラクティスである

・サービス => 顧客に価値を提供する手段
- コアサービス => 顧客が望む基本的な成果を実現するサービス : インターネット速度
- 実現サービス => コアサービスを提供する為に必要なサービス : 一定の通信速度
- 強化サービス => コアサービスをより魅力的にする為の、オプションサービス : 固定IPアドレス割り当て、無線LAN

- サービスオーナ => 特定のITサービスの提供について説明責任
- サービスマネージャ => ITサービスプロバイダ内のマネージャ

・サービスライフサイクル => ITサービスマネジメントをITサービスのライフサイクル(戦略、設計、移行、運用、継続的改善)の視点でまとめたもの
- サービスマネジメント => 顧客に対し、サービスの形で価値を提供する組織の専門的能力の集まり
- ITサービスマネジメント => 特にITにサービスを実施、管理すること

サービスマイ符サイクル
- サービスストラテジ(戦略) => 方針やサービスの位置付けを明確にし、ITサービスに対する戦略を立てる
- サービスデザイン(設計) => サービス戦略に沿って、ITサービスやプロセスを設計
- サービストランジション(移行) => 新規および変更されたITサービスを運用へ移行
- サービスオペレーション(運用) => 合意されたレベルでITサービスを提供する様に運用管理
- 継続的サービス改善(改善) => より良い戦略、設計、導入、運用を行える様に改善

・サービスプロバイダ
- タイプI - 内部サービスプロバイダ => 事業部門に内在するサービス・プロバイダ
- タイプⅡ - シェアード・サービス部門 => 複数の事業部門に共有ITサービスを提供
- タイプⅢ - 外部サービス・プロバイダ => 外部顧客にITサービスを提供するサービス・プロバイダ

・ITインフラストラクチャ => ITサービス提供に必要なハードウェア・ソフトウェア・ネットワーク・設備など
・サプライヤ => ITサービス提供に必要なITインフラストラクチャやITサービスをITサービス・プロバイダに提供する会社・組織

・プロセス => 特定の達成目標を実現するために設計された一連の活動
-インプットはデータや情報、アウトプットは期待される結果や成果物
・プロセスモデル => プロセス、プロセス・コントロール・プロセスの実現要因
- トリガ => プロセスの活動を開始する要因
プロセスの実現要因 => リソースと能力によって実現されることを意味する
- リソース => ITインフラストラクチャや人材、資金などの資産
- 能力 => ナレッジ(知識)や人材(経験、スキル)、組織などの無形資産を意味する

・プロセスの特性
- 測定可能性 => 効率的に運用できているか
- 特定の結果 => 個別に識別でき、数えられるもの
- 顧客 => 利害関係者に提供、期待に応えなければならない
- 特定のトリガへの応答性 => 特定のトリガに反応してプロセスは開始されるとゆう意味

・プロセスに関係する重要用語
- 機能 => 人材で構成されるチーム・グループと使用するツールやリソース
- 役割 => 個人・チームに与えられた責任・活動・職権。プロセス・機能内で定義
・RACIモデル => 役割定義する際、責任の所在を明確にし、組織の意思決定を支援
-- Responsible:実行責任者 => 業務実行に責任を持つ人(1人・複数)
-- Accountable:説明責任者 => 個々の活動の説明責任者(1人のみ)
-- Consuleted:協議先 => 相談を受け、意見を求められる人々(複数)
-- Informed:報告先 => 常に最新の進捗状況を受け取る人々(複数)
・プロセスの責任者
-- プロセス・オーナ => プロセスの成果に最終的な説明責任を負う
-- プロセス・マネージャ => プロセス運用管理に説明責任を負う
-- プロセス実務者 => 1つ以上のプロセス活動を実施することに責任を負う


★サービスストラテジ(戦略)

・サービスストラテジのプロセス
- サービス・ポートフォリオ管理 => プロバイダが管理する全サービスの情報の管理
- ITサービス財務管理 => ITサービス・プロバイダの予算・会計・課金・に対する要件管理
- 事業関係管理 => 顧客との良好な関係維持が責務

・サービス・ポートフォリオ => ITサービスの要件から廃止済みまでライフサイクル全体を管理
- サービス・パイプライン(検討中・開発中) => 顧客に提供できない全ITサービスの情報を格納
- サービス・カタログ(稼働中) => 導入準備中のITサービスの情報を格納
- 廃止済みサービス(段階停止・廃止) => 廃止対象サービスをカタログからいつ外すか、サービスレビューで各組織が決定

・価値創出 => サービスを通して顧客に価値を提供。サービス価値は有用性と保証の組み合わせで決まる
・有用性 => 特定のニーズを満たす製品・サービスによって提供される機能性
・保証 => 製品・サービスが合意された案件を満たすことを確約するもの
-- 必要な時に利用(可用性)、キャパシティ(受入れる能力)、継続性(状態を継続させる力)、セキュリティが提供されること

・サービス資産 => 顧客にサービスを提供するリソースと能力のこと
- リソース => サービス提供に必要な人材、情報、資金などの組織資産。時間小
- 能力 => ナレッジや人材、組織、プロセスなどの無形資産。時間大

・サービスの自動化 => サービスプロセスを自動化、有用性と保証を改善
- 自動化する前に、サービス・プロセスを簡略化
- 活動の流れ、タスクの割り当て、情報の必要性、やり取りを明確にする
- 基礎となるシステムやプロセスとユーザの接点の範囲を狭める
- 単純でも日常的でもないタスクとのやりとりを急がない

・ITサービス戦略管理 => 組織の観点・ポジション・計画・パターンを定義し維持するプロセス
・需要管理 => 顧客の需要を把握・予測し、影響を及ぼすプロセス

・事業関係管理とサービスレベル管理の違い
- 事業関係管理 => プロバイダがどのサービスを提供するかが焦点 => 顧客満足度の向上
- サービスレベル管理 => プロバイダがサービスレベルを達成できるかが焦点 => 合意されたサービスレベルの達成
※事業関係マネージャはサービスレベルマネージャを兼任することが多い。

・サービス指向のIT会計 => ITサービス毎にかかったコストを割り振り会計すること

・ITサービス財務管理の主要プロセス
- 予算業務 => 組織内の資金の収入と支出を予測し、コントロール
- 会計業務(IT会計) => ITサービス提供にかかるコスト分類・把握
- 課金 => 提供したITサービスの支払いを顧客に請求
・2つのサイクル
- 計画立案サイクル(年次) => 全体的支出に合意し、ITリソースんお単位コストを設定し、ITサービスの価格を設定
- 運用サイクル(月次・四半期) => 予算に対するコストを監視、請求書を発行し収益を回収

・ユーザ・プロファイル => ITサービスに対する要望パターン
・事業活動パターン => 顧客の事業活動パターン


★サービスデザイン(設計)

・サービスデザインのプロセス
- デザイン・コーディネーション =>全ての活動・プロセス・リソースを調整
- サービス・カタログ管理 => カタログを管理・維持し、利用を許可された人に提供
- サービスレベル管理 => 全てのITサービスが合意されたサービスレベルを満たす様に
- 可用性管理 => ITサービスの可用性に対し、費用対効果を満たしていく
- キャパシティ管理 => ITサービスとITインフラストラクチャに対し高い費用対効果を満たしていく
- ITサービス継続性管理 => サービスに深刻な影響を与える可能性のあるリスクを管理
- 情報セキュリティ管理 => 組織の資産・情報・データ・機密性・完全性・可用性が合意されたニーズに対応しているようにする
- サプライヤ管理 => 投資に見合う価値をサプライヤから取得できるように管理

・サービスデザインの5つの側面
- 新規または変更されるサービスに対するサービス・ソリューション
=> 事業部門と合意した要件にてサービスを設計
- 管理情報システムとツール(サービス・ポートフォリオ)
=> サービスマネジメントを支援する仕組みやツールを設計
- 技術アーキテクチャと管理アーキテクチャ
=> 技術は、インフラストラクチャ・アプリ・データのアーキテクチャを設計
=> 管理は、ITやサービスを管理するツールの利用方法や自動化の実現方法を設計
- 必要なプロセス
=> プロセスの活動内容・役割・責任・スキルを設計
- 測定方法や測定基準
=> サービスの品質やプロセス管理・評価する為の方法と基準を設計

・サービスデザインの4つのP
- People:人 => スタッフの能力やスキル
- Process:プロセス => サービス設計時の役割・活動
- Products:製品 => サービスや使用するツール・技術
- Partners:パートナ => サービス提供時に利用するサプライヤ・メーカ・ベンダ

〜〜〜キャパシティ管理のバランス => リソースとコスト・需要と供給
- キャパシティ計画 => ITサービスのリソース管理、予測し投資を計画
- キャパシティ管理のサブプロセス => 3つのサブプロセスがある
-- 事業キャパシティ管理 => 現在と未来の事業要件
-- サービスキャパシティ管理 => 事業をサポートするITサービスの提供
-- コンポーネント・キャパシティ管理 => ITサービスの提供を支えるITインフラストラクチャ

〜〜〜サプライヤ管理 => 契約上の義務を果たす
※サプライヤ…外部のメーカやサードパーティ
・サプライやのカテゴリ => 価値と重要性・リスクとインパクト
- 戦略的サプライヤ(上級マネ) => 戦略の機密情報を共有するパートナ関係(例：ネットワーク・サービス・プロバイダ)
- 戦術的サプライヤ(中級マネ) => 商業活動や事業でやり取りある(例：保守組織)
- 運用上のサプライヤ(運用マネ) => 運用上の製品やサービスを供給(例：利用頻度の低いウェブサイト運営)
- コモディティ・サプライヤ => 容易に入手できる製品やサービス(例：プリンタ用紙・トナーカートリッジ)
・契約 => 複数の当事者間における、法的な拘束力のある合意
- SLA => プロバイダと顧客との契約
- 外部委託契約(UC) => プロバイダとサードパーティ(外部サプライや)との契約
- サプライヤおよび契約管理情報システム(SCMIS) => サプライヤ管理の支援に使用される一連のツール・データ・情報
 => サプライや管理プロセスがSCMISを維持・管理

〜〜〜情報セキュリティ管理 => 情報セキュリティのCIA
- 機密性(Confidentiality) => 権限を持つ人のみ閲覧可能、情報やデータ保護
- 完全性(Intergrity) => 情報が正確で許可されていない修正から保護
- 可用性(Availability) => 必要な時に情報を入手・利用可能。攻撃に耐えられ、障害を防げる
・情報セキュリティ方針 => 組織のアプローチを統制する方針を明確にしたもの。
・情報セキュリティマネジメント(ISMS) => プロセス・機能・標準・指針・ツールの枠組み
 => ISO/IEC27001はISMSの国家規格
 
 〜〜〜ITサービス継続性管理
 - 事業継続性管理(BCM) => 事業に深刻な影響リスクを管理。達成目標・適用範囲・要件を設定
 - 事業継続性計画(BCP) => 中断後に回復に必要なステップを定義した計画、発動のトリガ・関与する人・コミュニケーションを特定
 - ビジネス・インパクト分析(BIA) => 中断するであろう事業へのインパクトを定量化。依存関係：サプライヤ・人材・ビジネス・プロセス・ITサービス
 => 目標復旧時間・目標復旧時点・最小限のサービスレベル目標値が定義
 
 ・リスクアセスメント => 脅威に対する各資産の脆弱性を評価すること
 => 定量的(数値を表す)・定性的(性質・特性を表す)ものがある。
 ・リスク管理 => リスクの識別・アセスメント・コントロール => 資産・脅威・脆弱性 => リスク => 対策
 ・復旧オプション
 - 手作業のワークアラウンド:ITサービスを利用しない、手作業による一時的な対策
 - 相互協定:類似技術を使用する組織間で非常時にリソースを共有
 - 段階的復旧(コールド・スタンバイ):場所やネットワーク配線を用意、ハードウェアの設置とソフトウェアのインストールを行う
 - 中間的復旧(ウォーム・スタンバイ):場所やハードウェアを用意、データのバックアップから復旧。1〜3日
 - 高速復旧(ホット・スタンバイ):場所とハードウェアとデータを用意、主系とデータを同期。24時間以下
 - 即時的復旧(ホット・スタンバイ):ミラー化、ロード・バランシング、サイト分割、綱に主系と待機系同期がとれる。即時再開可能
 
 ・ITサービス継続性管理と可用性管理の違い
 - 継続性 - 災害発生時に備える - 迅速にサービス復旧 - 非常事態に備えた計画・バックアップ
 - 可用性 - 日常の障害に備える - 可用性を高める - 対障害弾力性のあるシステム・バックアップ
 
 〜〜〜可用性管理
 - リアクティブな活動(反応) => イベント。インシデント・モニタリング・測定・分析・管理、運用上の役割
 - プロアクティブな活動(前向き) => 計画立案・設計・改善。計画立案の役割
 - コンポーネント可用性 => サービス構成するソフトウェア・ハードウェアなどのコンポーネント可用性
 - サービス可用性 => サービスそのものの可用性
 
 ・可用性の側面 => 可用性・信頼性・保守性・サービス性のモニタリング・測定・分析・報告を行う
 - 可用性(必要とされた時の実行能力) = (合意済みサービス期間-停止期間 / 合意済みサービス期間)×100
 - 信頼性(機能を実行し続けれる能力)
 -- 平均サービス・インシデント感覚(MTBSI)(時間) = 使用可能な時間/中断回数
 -- 平均故障間隔(MTBF)(時間) = 使用可能な時間-総停止時間/中断回数
 -保守性(障害発生後、稼働状態に戻す能力)
 -- 平均サービス回復時間(MTRS)(時間) = 総停止時間/サービス中断回数
 ※ MTBSI = MTBF + MTRS
 --サービス性(契約条件を満たす能力)

 ・重要事業機能(VBF) => 特に重要で不可欠な機能のこと
 
 〜〜〜サービスレベル管理(SLA:Agreement) => 顧客との合意
 - サービスレベル要件(SLR:Requirement) => 顧客の業務要件
 ・SLAの体系の設計 => 全てのサービス・顧客を網羅できる設計が必要
 - サービスベースのSLA => 単一サービスを規定、利用する全ての顧客にを対象
 - 顧客ベースのSLA => 特定の顧客グループと顧客が理想する全てのサービスを対象
 - マルチレベルSLA => 階層構造でSLAを管理、不必要な重複や頻繁な更新を回避できる
 -- 企業レベル => どの顧客に対しても一般的な内容。更新要求は頻繁にされない
 -- 顧客レベル => 企業レベルを引き継ぎ、サービスに関係なく、特定の顧客グループ・事業部門に対しての内容
 -- サービスレベル => 顧客レベルを引き継ぎ、特定の顧客に特定のITサービスに対しての内容
 
 - オペレーショナルレベル・アグリーメント(OLA) => プロバイダと同じ組織内の別の部署との間で交わす合意
 - 外部委託契約(UC) => プロバイダと外部サプライやとの間で交わす契約
 ※法的拘束力がある => 契約、　法的拘束力がない => 合意
 
 ・サービスレビュー => プロバイダと顧客が定期開催するMTG(毎月・四半期に一度)
 ・SLAMチャート(モニタリング)(RAGチャート) => サービスレベルを違反は赤、違反しそうは黄、達成は緑
 ・サービスレベル・マネージャ => SLAとOLAを文書化する責任
 ・サービス改善計画(SIP) => プロセス・サービスを改善する計画
 
 〜〜〜サービス・カタログ管理
 - 事業/顧客サービス・カタログ
 => 顧客に提供する全サービスの詳細、サービスに依存する事業部門とビジネス・プロセスの関係
 => 顧客が理解できる用語を使用
 - 技術/支援サービス・カタログ
 => 全支援サービスの詳細、支援するサービス・サービス提供に必要なコンポーネント・構成アイテム・その他支援サービスとの関係
 => 技術用語を使用
 
〜〜〜デザインコーディネーション
- 全体の設計活動 => 効果的・効率的に進めれる様に、設計活動の監督が必要 =>監督
=> デザイン・コーディネーション・プロセスのプロセス・マネージャによって行われる
①方針と手法の定義および維持
②設計におけるリソースと能力の計画
③設計活動の調整
④設計のリスクと課題の管理
⑤サービスデザインの改善

- 個々の設計活動 => PMかプロジェクトまたは変更に責任を持つ人が実施
❶個々の設計の計画
❷個々の設計の調整
❸個々の設計のモニタ
❹設計のレビューとサービスデザイン・パッケージ(SDP)の確実な引き継ぎ


〜〜〜サービストランジション
・サービストランジションのプロセス
- 移行の計画立案・サポート => 全てのサービストランジションプロセスの計画立案と必要なリソース調整
- 変更管理 => 変更のライフサイクルをコントロールし、ITサービスの中断を最小限に抑えながら、有益な変更を実施
- サービス資産・構成管理 => 必要な資産がコントロールされるようにする。資産に関して正確で信頼性のある情報を必要な時に利用できる
- リリース・展開管理 => リリースの構築・テスト・展開を計画立案・スケジューリング・コントロールすること。既存サービスの完全性を保護。新しい機能性も提供
- ナレッジ管理 => 観点・アイデア・経験・情報を共有し、適切な時期・場所で利用可能にする

・サービス資産・構成管理 => ITサービス構成情報を構成管理システム(CMS)として他のプロセス活動を支援
- 変更管理プロセス => 変更のインパクトを特定
- 可用性管理プロセス => 障害点を検出
- インシデント管理・問題管理プロセス => 問題が発生した機器の構成や診断情報を確認

・構成アイテム(CI) => 提供に必要なコンポーネントやサービス資産
・構成モデル => 構成アイテム間の関係を記録した、サービス・資産・インフラストラクチャのモデル
- インシデントと問題のインパクト・原因のアセスメント(評価)
- 提案された変更のインパクトのアセスメント
- 新規又は変更されるサービスの計画と設計
- 技術の更新・ソフトウェアのアップグレード計画
- リリースの計画・別の拠点やサービスセンタへのサービス資産の移動
- 資産の利用とコストの最適化(例：データセンタの統合・資産の再利用)
・構成管理データベース(CMDB) => CIの詳細であるレコードを格納するDB
=> サーバメンテナンスや問題が発生した時の影響を把握できる
・構成管理システム(CMS) => サービス・プロバイダの構成情報の管理に使用されるツール・DB。収集・保管・管理・更新・分析・提示するツールも含まれる
=> ITサービスマネジメントのプロセスに使用される
・確定版メディア・ライブラリ(DML) => 全てのソフトウェア構成アイテムの許可済みの確定版バージョンを安全に保存する1つ以上の場所
=> サービス資産・構成管理によってコントロールされる

・構成ベースライン => 聖騎士にレビューさ合意済みで、変更管理プロセスを通して管理される構成の基準。

・サービス資産・構成管理の5つの活動 (PICSV)
①管理と計画立案(Planning) => 構成管理する目的・方針・ツールを決定
②構成識別(Identification) => CIの管理・選択基準・命名規則を定義。CI収集後DM格納
③構成コントロール(Control) => CIの変更管理・許可済みCIのみ構成管理DMに登録できるようにする
④ステータスの説明と報告(Status) => 開発中・稼働中・使用中止、CI状態を追跡し構成ステータスの説明と報告を行う
⑤検証と監査(Verifacation) => 構成管理DBに登録れているかCIが存在するか、確認・監査

・変更管理
・変更管理の達成目標
- 顧客の変化する事業要件に対応・価値を最大化・インシデント・中断・手直し削減
- サービス・ニーズを整合、事業とITの変更要求に対応
- 変更を記録・評価し優先度をつけ、計画・テスト・実施・文書化・レビューする
- 構成アイテム全ての変更を構成管理システムに記録
- 全体的な事業リスクを最適化

・変更要求(RFC:Request For Change) => 変更の正式提案(紙・電子的に記録)
・変更要求の種類
- 標準的な変更 => リスクが低く、手順に従って行われる事前承認済み変更
- 緊急の変更 => サービスを復旧する場合等の迅速に導入しなければならない変更
- 通常の変更 => 標準的・緊急の変更でもない全てのサービス変更
※変更管理活動は、変更の実行者・変更許可委員・変更管理プロセス・オーナが実行

・変更提案 => 多大なコスト・リスク・インパクトが伴う重大変更の場合のみ使用される文書
- 事業の成果や提供できる有用性・保証を含む、変更するサービスの上位レベルの説明
- リスク・課題・代替手段・予算・財務上の期待事項を含めた美人エス・ケース
- 変更の設計と実施の大まかなスケジュール

・変更管理の7つのR
❶提起(Raised)
❷理由(Reason)
❸見返り(Return)
❹リスク(Risk)
❺リソース(Resource)
❻責任者(Responsible)
❼関係(Relationship)

・変更モデル => 特定のカテゴリ変更に対して繰り返し利用できる合意済み処理方法
- 課題や予想外のイベントの対処を含む、変更を処理するステップ
- 依存関係や一緒に処理するものの定義・ステップの実行順序
- 責任。誰が何を行うべきか(変更許可委員の特定を含む)
- 許容時間としきい値
- エスカレーション手順。誰にいつ連絡をとるか

・修復の計画立案 => 変更・リリースに失敗した場合に、復旧する処置
・変更諮問委員会(CAB) => 支援グループ。利害関係者の代表
・緊急変更諮問委員会(ECAB) => 変更マネージャにより招集されるCABより小さいグループ


・リリース・展開管理
・リリース => 構築・テスト・展開するサービス1つまたは複数の変更のこと
・リリース方針 => リリース活動の組織の方針
- リリースID・命名規則・期待されるリリースの頻度
- 役割と責任
- ベースラインを把握、リリース内容の検証方法
- リリーステスト・トレーニング・開始基準・終了基準・受け入れ職務権限
- 初期サポートを終了、引き継ぐ基準と許可

・リリース・ユニット => 同時リリースされるコンポーネント群
・リリース・パッケージ => 構築・テスト・展開される一連の構成アイテム
・リリース・展開管理4つの活動
①リリースと展開の計画立案
②リリースの構築とテスト
③展開
④レビューとクローズ

- Information:情報 => データを整理したもの(誰が、何を、いつ、どこで？)

・サービス・ナレッジ管理システム(SKMS) => ナレッジ・情報・データ管理のツール・DM

- 複数の移行を同時に管理する労力調整

〜〜〜サービスオペレーション
・サービスオペレーションの機能
- サービスデスク => プロバイダとユーザの単一窓口
- IT運用管理 => 管理必要な日々の活動を実施する機能
- 技術管理 => 管理において技術スキルを提供する機能
- アプリ管理 => アプリのライフサイクルを通した管理

・IT運用管理の目標
- 日々のプロセスと活動を安定させる
- 低コストでサービス向上、定常的な精査と改善
- IT運用障害を診断・解決する為の運用スキルの適用

・IT運用管理の活動・機能 => コントロールと施設管理
- IT運用コントロール => 日々の運用活動とイベントの監視、運用統制室・ネットワーク運用センタで行われる
-- 運用コントロールの活動
--- コンソール管理/運用統制室
--- ジョブ・スケジューリング
--- バッアップとリストア
--- 印刷と出力管理
--- 保守活動
- 施設管理 => 物理的なIT環境を管理

・サービスオペレーションのプロセス
- イベント管理 => インフラストラクチャで発生するイベントのライフサイクル管理
- インシデント管理 => インシデント(計画外・中断・品質)のライフサイクル管理
- 要求実現 => サービス要求(ユーザ=>組織)のライフサイクル管理
- 問題管理 => 問題(インシデントの根本原因)のライフサイクル管理
- アクセス管理 => 許可済みユーザに権限付与、無許可の場合、アクセス権限

・アクセス管理の目標
- 情報セキュリティ管理で定義された方針と処置でアクセス管理
- アクセス権の付与・変更・制限に対応し、適切に付与される
- アクセス監視・権限が不適切に使用されないようにする

・アクセス => ユーザーに使用権限があるサービスの機能やデータレベル・範囲
・ID => ユーザを個人として区別ユーザに関する情報
・権限 => サービス群へアクセス提供する際の設定の事。読み・書き・実行・変更削除がある。
・サービス群 => 複数のサービスの集まり、サービスグループ
・ディレクトリ・サービス => アクセスや権限管理専門のツール(OpenLDAP,Active Dirextroy)

・要求実現の目標と活動
- 顧客満足度の維持
- 許可と審査に必要な窓口をユーザに提供(PWリセット・新規PCの設置)
- サービスの可用性・手順の情報を提供(アプリ使用方法を教える)
- サービスのコンポーネントを調達・提供(プリンタのトナー交換・ライセンス発行)
- 情報・苦情・コメント対応

・要求モデル => 特定の種類のサービス要求に対する処理方法
・セルフヘルプ => ユーザ自らサービス要求やインシデント登録できる仕組み

・アプリケーション管理 => サービスマネジメント活動を支援・管理サポートする部門・グループ・チームによって実施される
- アプリケーションの設計: 対障害弾力性と費用対効果の高いアプリ設計
- 機能の提供: 事業成果の達成に必要な機能を利用できるようにする
- アプリの保守: 最適な状態に維持
- サポート: 障害を診断士解決する技術力

・アプリケーション管理の組織 => 財務・人事・メッセージング・アプリ・事業固有アプリにカテゴライズされる
・アプリケーション開発 => 要件・設計・構築の為の一回限りの活動。管理チームとは異なるチームとして機能する

-開発チーム
--性質: 設計・構築する為の一回限りの活動
--対象: 内部開発。アプリに対し実施。
--焦点: 有用性重視。アプリが何を行うか
--管理方法: 開発作業は特手のまとまりのプロジェクト内で行う。次のプロジェクトに入るとサポートが困難
--コスト: リソースが明確で定量化しやすい、費用を関連付けられる

-管理チーム
--性質: ライフサイクル全体の管理、継続的な活動
--対象: 購入・内部開発問わず、全てのアプリ
--焦点: 有用性・保証を重視、機能性が何か、どのように提供するか
--管理方法: 反復可能で継続的なプロセスの一環、運用スタッフは開発に関与しない
--コスト: 共有されることが多く、管理コストが他のサービスと一緒にされる

・技術管理の活動
- ITインフラストラクチャの設計: 対象外弾力性と費用対効果の高いインフラストラクチャ設計
- ITインフラストラクチャの保守: 最適な状態に維持する技術力提供
- サポート: 発生した象外を診断し、解決する技術力提供

・技術管理の組織 => 単一ではなく、インフラストラクチャの種類毎に個別のチームや部門が担当

・サービスデスク
・サービスデスクの活動
- インシデント・サービス要求の記録、カテゴライズと優先付け
- 1次調査と診断
- 初回問い合わせ時の解決
- 合意された期間内に解決できない際のエスカレーション
- ユーザへの進捗状況報告
- 解決済みのインシデント・サービス要求・コールのクローズ
- 顧客満足度調査の為の、コールバック
- ユーザとのコミュニケーション
- サービス資産・構成管理の指示と承認の下で構成管理システムの更新

・サービスデスクの構造
- ローカル・サービスデスク - ユーザの拠点毎に設置 - オンライン対応・拠点の言語や文化に沿った、きめ細かいサービス
- 中央サービスデスク - 複数の拠点を1ヶ所・少数の窓口に統合 - コストを抑え、少ないスタッフで対応、スタッフのスキル向上
- バーチャル・サービスデスク - 複数の拠点に設置された窓口を1つの中央サービスデスクのように運用 - 在宅勤務・2次サポート・グループアウトソーシングを組み合わせて柔軟に対応・24時間サポート

-フォロー・ザ・サン(24時間サポート) => 各拠点のサービスデスクの時差を利用して24時間サポートを実現する方法


・問題管理
・問題管理の目標
- 問題と結果によるインシデントの発生を防ぐ
- 再発するインシデントの排除
- 帽子できないインシデントのインパクトを最小限に抑える
--問題管理とインシデント管理の違い
問題管理=> 根本原因の特定・解決・予防
インシデント管理=> サービス運用に迅速に回復させる
 
・問題モデル => 特定の種類の再発する問題を処理するステップを定義する方法
・ワークアラウンド => まだ完全な解決策が存在しないインシデントや問題のインパクト低減、排除すること。「一時的な回避策」
・既知のエラー => 文書化された根本原因とワークアラウンドのがある問題
・既知のエラーDB(KEDB) => 問題管理が作成・管理、インシデント・問題管理が使用する。構成管理システム(CMS)の一部とサービス・ナレッジ管理システム(SKMS)の任意の場所にも格納可能

・問題管理の活動 => プロアクティブ・リアクティブ問題管理
- プロアクティブな問題管理 => 過去のレコードを分析し、傾向を識別することで、インシデントの発生を予防する
- リアクティブな問題管理 => インシデントが発生してから分析し、根本原因を特定する
・リアクティブな問題管理の活動プロセス
①問題の検出(イベント・インシデント管理)
②問題の記録
③問題のカテゴリ化
④問題の優先度付け
⑤問題の調査と診断
⑥ワークアラウンドの特定
⑦既知のエラー・レコード提起(有益となった場合)
⑧問題の解決
⑨問題のクローズ
⑩重大な問題のレビュー


・インシデント管理
・インシデント管理プロセスへの報告
- イベント管理プロセスからエスカレーション
- 顧客・ユーザからの報告(サポートデスク経由)
- 管理ツールからの通知
- 技術スタッフからの報告

・インシデントモデル
- インシデントに対応する為のステップ
- 依存関係があるものや同時に処理すべきものの定義・時系列の処理順序
- 責任(担当者、担当内容)
- インシデントを解決する前にとるべく予防策(データファイル・バックアップ)
- 処置を完了する為の許容時間としきい値
- エスカレーション手順
- 証拠保全の活動

・重大なインシデント => 事業にとって深刻な中断の原因になるインシデント、短い時間での対処の為、個別のインシデント・モデルを用意する
・インシデントの優先度付け => 優先度付けのコードを事前に顧客と合意が必要
・インパクト => 影響を及ぼす指標。サービスレベルが受ける影響の程度
- 生命や健康に関わるリスク
- 影響を受けるサービス数
- 財政上の損失のレベル
- 事業の評判への影響
- 規則上・法律上の違反

・緊急度 => インパクトを与えるまでに残された時間の指標
・優先度 => 相対的な重要度の識別。事前にSLAへ定義。

・インシデントのエスカレーション => サービスレベルや顧客の期待を満たす為、必要なリソースを入手する活動
- 機能的エスカレーション => 単独解決できない場合・紹介解決時間を超えた時点、専門知識グループにインシデントを渡す
※上位サポート・グループを「2次・3次サポート」と言う
--1次サポート：最初にインシデントを受けるサポートデスク
--2,3次サポート:より多くの時間とリソースを所有するサポート・グループ

-階層的エスカレーション => 優先度の高いインシデント、調査や解決に長時間かかる場合、権限を持つ上位マネジメントに情報提供したり、関与させる

-インシデント管理の活動
❶インシデントの識別(サービス要求->要求実現、変更提案->サービス・PF管理)
❷インシデントの記録
❸インシデントのカテゴリ化
❹インシデントの優先度付け(インパクトと緊急度)
❺初期診断
❻インシデントのエスカレーション(機能的か階層的のエスカレーション実施)
❼調査と診断(既知の回避策がない場合)
❽解決と復旧(サポートデスク・専門業者が解決)
❾インシデントのクローズ(サポートデスクが、進捗の追跡・ユーザへの情報提供・クローズまでが責任)


・イベント管理 => イベント管理ツールを使用しインフラストラクチャで発生するイベントを監視・検出
・イベント => 構成アイテムやサービスの管理にとって重要性のある状態の変更
・イベントの種類
- 情報イベント - 対処を必要としない - バッチ処理が正常、アプリにログイン
- 警告イベント - 以上を示し、注意必要 - 許容範囲上限80%の内75%に達した
- 例外イベント - 正常に運用できない状態 - 不正PWでログイン、サーバが停止

・イベント管理の活動
- イベントには「サービスデスク」・「IT運用管理」・「技術管理」・「アプリ管理」のサービスオペレーションの機能が対応します。
- 自動応答 => 処置が明確で自動化できるもの。例：サーバ再起動・スクリプト実行
- アラート => 人の介入が必要なイベント、スキルを持った人に通知すること

・イベント管理ツールの種類
- 能動的モニタリング・ツール => 主な構成アイテムをポーリング、ステータスと可用性を判断
- 受動的モニタリング・ツール => 構成アイテムが一定の条件に達する際にアラームやイベントを検出。SNMPを使用してイベント生成

・サービスオペレーションにおけるコミュニケーションの役割
- 運用に関する日養生的なコミュニケーション：提起的な活動を調整。通常運用に影響がある可能性がある変更を認識する
- シフト間のコミュニケーション：引き継ぎのこと
- パフォーマンス報告
- プロジェクトにおけるコミュニケーション:作業の割り当て・プロジェクトのホウレンソウ
- 変更に関するコミュニケーション:変更管理、リリース・展開管理をサポート
- 例外に関するコミュニケーション:事象が発生した際の情報共有
- 緊急事態に関するコミュニケーション:インパクトと重大度を調査し、緊急事態を確認
- グローバルなコミュニケーション
- ユーザ・顧客とのコミュニケーション:最新情報の提供。技術的な説明や内部プロセスの情報は含めません。
